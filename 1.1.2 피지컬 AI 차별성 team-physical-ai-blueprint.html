◆ 02 Hardware Combo

왜 SO-ARM101 + LeKiwi 조합인가

값비싼 산업용 로봇 한 대보다, 저비용 오픈 하드웨어 두 종이 교육적으로 더 강력합니다.
각자 다른 제약과 다른 학습 포인트를 만들어내기 때문입니다.

🦾
SO-ARM101
Demonstration · 정밀 Manipulation 트랙
구조적 특징
Leader / Follower 구조 + 기어 구성이 달라 사람이 시범을 보이면서 데이터를 수집하기 좋은 설계
교육 포인트
Teleop 기반 demonstration 수집 · 정밀 manipulation을 직접 손에 익히면서 배움
학습 도달점
"데이터의 질이 모델의 질을 결정한다"는 명제를 몸으로 체득하게 됨
🛞
LeKiwi
Edge 제약 · Async Inference 트랙
구조적 특징
Raspberry Pi 또는 5V 구동 PC를 실어 운용하도록 설계 — edge 제약을 일찍 드러내는 플랫폼
교육 포인트
모바일 + 제한된 컴퓨트 환경에서 어떤 모델이 작동하고 어떤 모델이 멈추는지 직접 확인
학습 도달점
Edge / Cloud 분할 · Async Inference의 필요성을 자연스럽게 깨닫게 됨
💡 SmolVLA 공개 자료도 SO-ARM101, LeKiwi 같은 저비용 오픈 하드웨어를 실험 대상으로 명시하고 있습니다. 따라서 우리 교육의 저비용 — 고학습효과 메시지와 정확히 정합합니다.

◆ 04 Model–Device Fit

모델 비교의 진짜 목적

ACT · Diffusion · SmolVLA · pi₀-fast · WorldVLA를 다 다루지만,
"어느 모델이 짱이냐"가 아니라 "어떤 상황에 어떤 모델이 맞느냐"를 가르칩니다.

// THINKING FRAME
"어떤 모델이 최고냐?"
→ "어떤 공정 · 어떤 로봇 · 어떤 디바이스 · 어떤 지연 예산에서
어떤 모델이 더 맞는가?"
🦾
SO-ARM101 트랙
정밀 manipulation · 고정 작업대
ACT → Diffusion → SmolVLA
첫 baseline은 ACT가 가장 좋습니다. 이후 Diffusion · SmolVLA로 올라가며 교육생이 "precision vs compute" 트레이드오프를 몸으로 이해하게 만드는 흐름이 가장 효과적입니다.
🛞
LeKiwi 트랙
모바일 · 제한된 edge 컴퓨트
SmolVLA + pi₀-fast
Perception–Language–Action과 Async / Remote Inference를 함께 보여주기 좋은 조합. 모바일 플랫폼에서는 모델 정확도만큼이나 큐 대기 시간 · 재계획 반응성 · 네트워크 추론 구조를 같이 평가해야 합니다.

◆ 05 Async Inference

왜 Async / Remote Inference를 가르치는가

큰 모델(bulky policy)을 모바일에서 그냥 돌리면, 로봇이 추론을 기다리는 동안 멈춰있습니다.
이 문제를 해결하는 구조 자체가 교육의 핵심 포인트입니다.

▼ Sync · Naïve

순차 추론 — 로봇이 기다린다

act

▲ Async Stack

비동기 추론 — 동시에 진행

act
추론과 행동이 파이프라인처럼 겹쳐 진행. 로봇은 멈추지 않고 계속 움직임.
≈ 2× 단축
SmolVLA async stack 적용 시 task completion time이 약 2배 개선된 사례 — 모델을 바꾸지 않고 실행 구조만 바꿔서 얻어지는 성능

따라서 모바일 플랫폼 평가는 정확도 외에 다음 4가지 지표를 함께 봅니다 —

⏳
Queue Starvation
로봇이 명령을 기다리는 시간
🔁
Replanning Responsiveness
상황 변화 시 재계획 반응 속도
🌐
Networked Inference
네트워크 추론 안정성·지연
◆ 06 Roadmap

우리는 이렇게 추진합니다 — 5단계 청사진

각 단계는 이전 단계의 결과물을 입력으로 받아 다음 단계로 이어집니다.
팀 프로젝트 하나가 5단계를 모두 관통합니다.

S1
현장 문제를 데이터 구조로 번역 G/01 · L1–L2
팀이 자기 공장의 문제를 골라, 그것을 input · state · action · reward 로 풀어 씁니다. 여기서 demonstration · telemetry 수집 계획을 함께 설계합니다.
문제 정의서 데이터 수집 계획
S2
Teleop Demonstration 수집 + Baseline 학습 G/02 · L2–L3
SO-ARM101에서 leader/follower로 직접 시범을 보이며 데이터 수집. ACT를 baseline으로 학습 — 데이터 질이 결과에 어떻게 반영되는지 관찰합니다.
Teleop 데이터셋 ACT Baseline
S3
모델 비교 · 모델 ↔ 디바이스 적합성 판단 G/03 · L3
ACT → Diffusion → SmolVLA로 올라가며 SO-ARM101에서 비교. LeKiwi에서는 SmolVLA · pi₀-fast로 모바일 환경에서 무엇이 다른지 직접 측정합니다.
모델 비교 리포트 적합성 판단 근거
S4
Edge / Cloud 분할 + Async Inference + 프로토콜 연동 G/04 · G/05 · L4–L5
어디서 추론하고 어디로 보낼지 결정. MQTT · UNS · OPC UA로 공정 시스템과 연동하고, 지연·안전 예산을 만족하는 시스템 구조를 설계합니다.
시스템 아키텍처 Async 파이프라인 프로토콜 연동
S5
KPI 연결 발표 — 경영 언어로 번역 G/06 · L6
기술 결과를 사이클 타임 · OEE · 양품률 · 안전 지표로 번역하여 발표. "AI를 도입하면 무엇이 얼마나 좋아지는가"를 경영진이 결재할 수 있는 형식으로 정리합니다.
KPI 리포트 발표 자료 투자회수 시뮬레이션
